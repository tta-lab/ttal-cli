This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A Wails v3 desktop app (`ttal-gui`) — Go backend + SvelteKit frontend. The GUI is a chat interface for talking to ttal agents. It reads from `~/.ttal/messages.db` (SQLite via Ent) and sends messages through the ttal daemon socket.

## Commands

```bash
# Dev mode (hot-reload for both Go + Svelte)
wails3 dev -config ./build/config.yml

# Build production binary
wails3 build

# Regenerate TypeScript bindings after changing ChatService methods
wails3 generate bindings

# Frontend only (from gui/frontend/)
npm run dev
npm run build
```

All task shortcuts are in `Taskfile.yml` — run with `task dev`, `task build`, etc.

## Architecture

### Backend (`app.go`)

`ChatService` is the only Wails service. Every exported method on it becomes a callable TypeScript function. It has two data paths:

- **Reads**: query SQLite directly via Ent (`s.db`)
- **Writes**: go through daemon unix socket (`~/.ttal/daemon.sock`) as a `SendRequest` JSON blob — this keeps delivery semantics consistent with CLI `ttal send`

`main.go` wires it up: loads `~/.config/ttal/config.toml` via `config.LoadAll()`, constructs `ChatService`, registers it as a Wails service, opens a single window.

### Frontend (`frontend/src/`)

- **`routes/+page.svelte`** — single page, owns all polling intervals and global keyboard handlers
- **`lib/stores/messages.svelte.ts`** — Svelte 5 class-based reactive store (`chatStore`), single source of truth for messages, contacts, active contact/tab
- **`lib/components/`** — `Sidebar`, `ChatBubble`, `MessageInput`, `ReactionBar`, `AgentFeed`

Polling strategy: contacts every 5s, messages every 500ms when a contact is active, feed every 500ms when feed tab is active.

### Bindings (`frontend/bindings/`)

Auto-generated by `wails3 generate bindings`. The ttal-specific bindings live at:
```
frontend/bindings/github.com/tta-lab/ttal-cli/gui/
  chatservice.js   — all ChatService methods
  models.js        — Go struct types (Contact, etc.)
```

After adding/renaming any exported `ChatService` method or struct, regenerate bindings before touching the frontend.

### Data model

Messages are stored in `~/.ttal/messages.db`. The Ent schema is in `../internal/ent/schema/`. Key fields: `sender`, `recipient`, `content`, `created_at`, `team`.

Human identity comes from `[user] name` in config.toml (falls back to `$USER`). This name is used to split messages into "mine" vs "theirs" in queries.

### Avatar loading

`GetAvatar(agentName)` searches all teams' `team_path` directories for `{agent_path}/assets/avatar.{png,jpg,jpeg}`. The frontend caches blob URLs in `avatarCache` (Map) and only fetches once per agent per session.

## Key constraints

- Do NOT import `internal/daemon` in the gui package — it pulls in CGO-incompatible watcher deps. Mirror any shared types locally (see `SendRequest` in `app.go`).
- The gui has its own `go.mod` — it is a separate Go module from the root `ttal-cli` module, but imports from `../internal/` via a `replace` directive.
